## 1. 비즈니스 전략 개요

0to1log는 AI 뉴스 콘텐츠 플랫폼이자 **Global-to-Local Insight Bridge**다. 비즈니스 전략은 이 정체성에 맞춰 **콘텐츠 → 트래픽 → 수익**의 순서로 설계한다.

핵심 원칙:

- **오가닉 퍼스트:** SEO와 바이럴로 유기 트래픽을 먼저 확보한다. 유료 광고 집행은 수익 발생 후.
- **단계적 수익화:** 무료 콘텐츠로 신뢰를 쌓은 뒤, 광고 → 프리미엄 구독 순서로 수익화한다.
- **데이터 기반 의사결정:** GA4와 MS Clarity로 유저 행동을 수집하고, AARRR 지표로 그로스를 측정한다.
- **EN canonical 운영:** EN을 기준 원본으로 두고 KO는 localized derivative로 운영한다.
- **검증 후 확장:** PWA로 앱 경험을 검증한 뒤에만 네이티브 앱을 개발한다.

## 2. 수익화 로드맵

### 2.1 수익 모델 전체 구조

```
Phase 1~2          Phase 2 후반         Phase 3~4            Phase 5
──────────         ─────────           ──────────           ──────────
트래픽 축적    →    AdSense 신청   →    프리미엄 구독 검토  →   인앱 결제
(SEO + 바이럴)      (광고 수익)         (콘텐츠 계층화)        (Polar)
```

> **ADR — 왜 이 순서인가:** Solo 프로젝트에서 수익 전에 비용을 쓰면(유료 광고, 네이티브 앱 개발) 리스크가 크다. 비용 제로인 SEO/바이럴로 트래픽을 먼저 확보하고, 트래픽이 증명된 후 수익화를 시작한다.

### 2.1.1 정량 게이트 (Go / Hold)

의사결정을 감으로 하지 않기 위해, 핵심 전환 포인트를 정량 KPI로 고정한다.

| 의사결정 | Go 조건 (윈도우 태그 포함) | Hold 조건 |
|---|---|---|
| **AdSense 신청 [28D]** | 게시글 30개 이상 + 오가닉 세션 3,000+ + 평균 engagement time 90초+ + 재방문 사용자 비율 20%+ | 위 조건 미달 |
| **프리미엄 검토 시작 [8W-MA]** | 최근 8주 이동평균 WAU 500+ + 재방문 사용자 비율 30%+ + 페르소나 전환 사용률 15%+ | 위 조건 미달 |
| **프리미엄 출시 [28D]** | 웨이트리스트 100명+ + paywall 대상 글 완독률 35%+ + 결제 전환율 2%+ | 위 조건 미달 |
| **네이티브 앱(Phase 5) 진입 [4W-Cohort]** | PWA 설치율 4%+ (4주 연속) + 설치자 4주 유지율 25%+ + 푸시 opt-in 35%+ | 위 조건 미달 |

> 윈도우 태그: `[28D]=최근 28일`, `[8W-MA]=최근 8주 이동평균`, `[4W-Cohort]=4주 코호트 기준`.

### 2.1.2 인프라 비용 상향 전략 (Railway)

| 단계 | 운영 모드 | 비용 관점 | 비즈니스 연결 KPI |
|---|---|---|---|
| **Stage A (현재)** | 범용 API 구조 + 저비용 운영 | 비용 최소화, 기능 검증 우선 | `API<300/일` 또는 `검색<80/일` 또는 `커뮤니티<40/일` 또는 `재방문<25%` |
| **Stage B (Always-on)** | 상시 API 운영 | 기본료 + 컴퓨팅 변동비 반영 | 최근 28일 기준 `API 300+/일 + 검색 80+/일 + 커뮤니티 40+/일 + 재방문 25%+` |

> **원칙:** 비용 상향은 "기술 선투자"가 아니라 KPI 달성 이후 "수요 확인된 증설"로 진행한다.
> **Go/Hold 계약:** 위 4개 지표가 모두 충족되면 Go(Stage B), 하나라도 미달이면 Hold(Stage A 유지).

### 2.2 AdSense (Phase 2 후반)

**시작 조건:**

- 콘텐츠 30개 이상 축적
- 최근 28일 오가닉 세션 3,000+
- 평균 engagement time 90초+
- 재방문 사용자 비율 20%+

**현실적 기대치:**

- Solo 블로그 AdSense 수익은 월 수만 원 수준이 한계다
- 0to1log의 타겟(개발자/AI 관심층)은 일반 디스플레이 광고 CPM이 낮은 편
- 이 단계는 "수익화 경험"을 확보하는 것이 주 목적

**구현 고려사항:**

- AdSense 스크립트가 CSP(Content Security Policy)에 영향을 줌 → `05_Infrastructure.md`에서 관리
- 광고 배치: 콘텐츠 하단 또는 사이드바, 읽기 경험을 해치지 않는 위치

**Phase 3~4 대안 검토:**

- 스폰서 콘텐츠 / 뉴스레터 광고: 개발자 대상 CPM이 AdSense보다 높음
- 이 대안은 트래픽과 브랜드 인지도가 충분히 쌓인 후 고려

### 2.3 프리미엄 구독 모델 (Phase 4 검토)

**핵심 질문: "뭘 유료로 제공할 것인가?"**

0to1log에서 유료화 가능한 콘텐츠 구조:

| 계층 | 접근 범위 | 대상 |
|---|---|---|
| **무료** | Research 포스트 전체 + Business 포스트 (비전공자/학습자 버전) | 모든 방문자 |
| **프리미엄** | Business 포스트 Expert 버전 + 심층 분석 + 아카이브 검색 | 구독자 |

**전제 조건:**

- 무료 콘텐츠로 신뢰와 품질이 증명되어야 함
- "이 콘텐츠는 돈 내고 볼 가치가 있다"는 인식이 먼저 형성되어야 함
- 최근 8주 이동평균 WAU 500+ / 재방문 사용자 비율 30%+
- 페르소나 전환 사용률 15%+ / 공유 이벤트율 1%+ (세션 기준)

**결제 연동: Polar**

- Stripe 대비 셋업이 간단하고, 개인 개발자에게 친화적
- 구독 모델이 확정되면 연동 (Phase 4 후반 ~ Phase 5)
- Supabase RLS 정책 변경 필요: 구독 상태에 따른 콘텐츠 접근 제어

> **ADR — 왜 구독 모델을 Phase 4까지 미루는가:** 신뢰 없이 유료화하면 이탈만 가속된다. 무료 콘텐츠로 6개월 이상 운영하며 "이 블로그의 분석은 다르다"는 평가를 먼저 확보한다.

### 2.4 퍼포먼스 마케팅 (Phase 4 이후 조건부)

- Meta, Google 유료 광고 집행은 **수익이 발생한 후**에만 고려
- Solo 프로젝트에서 수익 전 광고비 지출은 리스크가 큼
- 우선순위: SEO + 바이럴 → 오가닉 트래픽 확보 → 수익 발생 → 그 후 유료 광고

> **ADR — 유료 광고를 뒤로 미루는 이유:** 오가닉 트래픽이 확보되지 않은 상태에서 광고로 유입시킨 사용자는 리텐션이 낮다. 콘텐츠와 SEO로 자연 유입 구조를 만든 뒤, 유료 광고는 부스터 역할로만 사용한다.

## 3. 데이터 분석 체계

### 3.1 분석 도구 스택

| 도구 | 설치 시점 | 역할 | 비용 |
|---|---|---|---|
| **Vercel Analytics** | Phase 1 (기 계획) | 페이지 성능, Web Vitals | 무료 (Hobby) |
| **GA4** | Phase 1b | 트래픽 소스, 유저 흐름, 전환 추적 | 무료 |
| **MS Clarity** | Phase 1b | 히트맵, 세션 리플레이, 분노 클릭 감지 | 무료 |

**설치가 Phase 1b인 이유:**

- 스크립트 한 줄이면 설치 완료 — 기술 부담 거의 없음
- 데이터는 일찍 쌓기 시작할수록 나중에 의미 있는 분석이 가능
- Phase 3에서 AARRR 대시보드를 만들 때 최소 3~4개월 데이터가 필요

**인프라 영향:**

- GA4: `connect-src`에 `https://www.google-analytics.com` 추가
- Clarity: `connect-src`에 `https://www.clarity.ms` 추가
- 상세 CSP 설정은 → `05_Infrastructure.md` 참조

### 3.2 AARRR 지표 프레임워크 (Phase 3)

GA4 데이터가 충분히 쌓인 후, AI Ops Dashboard에 그로스 지표를 시각화한다.

| 지표 | 측정 항목 | 데이터 소스 |
|---|---|---|
| **Acquisition (유입)** | 일일 방문자, 트래픽 소스별 비율, 검색 유입 키워드 | GA4 |
| **Activation (활성화)** | 페르소나 설정 완료율, 첫 포스트 끝까지 읽은 비율 | GA4 + Clarity |
| **Retention (유지)** | 주간 재방문율, 세션 깊이 (2페이지+ 방문 비율) | GA4 |
| **Revenue (매출)** | AdSense 수익, 프리미엄 전환율 (Phase 4+) | AdSense + Polar |
| **Referral (추천)** | SNS 공유 횟수, Highlight to Share 사용률 | 커스텀 이벤트 |
| **Technical (운영)** | 검색 요청량, 커뮤니티 액션량, API 요청량/오류율 | FastAPI 로그 + Railway + Supabase |

**구현 방식:**

- AI Ops Dashboard (Phase 3) 내에 AARRR 탭으로 통합
- GA4 Reporting API로 데이터를 가져와 Supabase에 캐싱 → 프론트엔드에서 차트 렌더링
- 복잡한 BI 도구(Looker, Metabase)는 사용하지 않음 — 자체 대시보드로 충분

### 3.3 KPI 계산 정의 (고정)

| KPI | 정의 |
|---|---|
| **재방문 사용자 비율** | `returning_users / total_users` |
| **평균 engagement time** | GA4의 `user_engagement` 기준 평균 참여 시간 |
| **페르소나 전환 사용률** | `persona_switch 이벤트 발생 세션 / 전체 세션` |
| **공유 이벤트율** | `share 이벤트 발생 세션 / 전체 세션` |
| **PWA 설치율** | `설치 완료 수 / 설치 프롬프트 노출 세션` |
| **설치자 4주 유지율** | `설치 주차 +4주에도 활성인 설치자 / 설치 주차 설치자` |
| **결제 전환율** | `유료 구독자 수 / paywall 노출 사용자 수` |
| **MRR** | `해당 월 반복 구독 매출 합` |
| **churn rate** | `해당 기간 이탈 유료 구독자 수 / 기간 시작 유료 구독자 수` |
| **refund rate** | `해당 기간 환불 건수 / 해당 기간 결제 건수` |
| **paid retention (4주)** | `결제 주차 +4주에도 활성인 유료 구독자 / 결제 주차 신규 유료 구독자` |
| **검색 사용량** | `semantic_search 요청 수 / 일` |
| **커뮤니티 액션량** | `points_earn + prediction_join 이벤트 수 / 일` |
| **API 요청량** | `전체 FastAPI 요청 수 / 일` |
| **Cross-lingual Recall@K** | 한국어/영어 쿼리 상호 검색 시 관련 문서 회수율 |
| **Locale-aware nDCG@K** | locale 우선순위를 반영한 검색 정렬 품질 |
| **Language-mismatch bounce** | 언어 불일치 결과 클릭 후 즉시 이탈 비율 |

> 후속 확장 항목: `ARPU`, `LTV`, `CAC`, `Payback`은 Phase 4 이후 데이터 안정화 시점에 추가한다.

### 3.3.1 KPI 집계 윈도우 규칙 (계약)

- 기본 의사결정 윈도우: `최근 28일 [28D]`

- 예외 2 (PWA/유료 유지): `4주 코호트 [4W-Cohort]`
- 게이트 표/마일스톤 표에는 반드시 윈도우 태그를 함께 표기한다.

### 3.4 기술 KPI ↔ 비즈니스 KPI 연결

## 4. SEO / GEO 전략

### 4.1 SEO 기반 설계 (Phase 1a — 최우선)

**왜 Phase 1a에서 해야 하는가:**

Astro는 SSG 기반이라 SEO에 기본적으로 유리하다. 하지만 구조화 데이터(JSON-LD), 사이트맵, 메타 태그 최적화는 콘텐츠 생성 전에 설계해야 한다. 나중에 적용하면 기존 콘텐츠 전체를 수정해야 하기 때문이다.

**Phase 1a 구현 항목:**

| 항목 | 설명 | 구현 위치 |
|---|---|---|
| **JSON-LD 구조화 데이터** | 뉴스 기사 스키마 (`NewsArticle`, `Person`, `WebSite`) | Astro 레이아웃 |
| **사이트맵 자동 생성** | Astro 빌트인 사이트맵 생성 + robots.txt | `astro.config.mjs` |
| **i18n SEO 태그** | `hreflang(ko/en/x-default)` + self-canonical | 공통 `<Head>` 컴포넌트 |
| **메타 태그 최적화** | `<title>`, `<meta description>`, Open Graph, Twitter Card | 공통 `<Head>` 컴포넌트 |
| **시맨틱 HTML** | `<article>`, `<time>`, `<nav>` 등 적절한 시맨틱 태그 사용 | 모든 페이지 |
| **URL 구조** | locale-aware slug: `/en/log/[slug]`, `/ko/log/[slug]` | Astro 라우팅 |

**AI NEWS 포스트 SEO 특화:**

- 매일 발행되는 AI NEWS는 검색 유입의 핵심 자산
- 뉴스 키워드를 제목과 slug에 반영 — AI 파이프라인의 Editorial 에이전트가 자동 처리
- `NewsArticle` JSON-LD에 `datePublished`, `author`, `publisher` 필수 포함

> 상세 프론트엔드 구현은 → `04_Frontend_Spec.md` 참조

**기본 진입 언어 정책 (EN-first):**

- 루트 및 기본 진입은 EN을 기준으로 운영한다.
- `x-default`는 `/en/`을 가리킨다.
- KO 페이지는 EN 기준 콘텐츠의 localized 버전으로 운영한다.

### 4.2 GEO — 생성형 AI 검색 최적화 (Phase 3)

GEO(Generative Engine Optimization)는 기존 SEO를 넘어, AI 검색 엔진(Google AI Overview, Perplexity 등)에서 0to1log 콘텐츠가 인용되도록 최적화하는 전략이다.

**Phase 3 적용 항목:**

- 질문-답변 형식의 콘텐츠 섹션 추가 (AI가 추출하기 쉬운 구조)
- 명확한 정의와 수치 데이터를 본문에 포함 (인용 확률 증가)
- 콘텐츠 요약(TL;DR)을 포스트 상단에 배치
- 출처 링크를 명시적으로 표기 (AI 검색의 신뢰도 신호)

**GEO 성공 기준 (월간, 정량):**

| 지표 | 정의 | 초기 기준 |
|---|---|---|
| **AI 검색 유입 세션(월)** | AI 검색 엔진/어시스턴트 유입 세션 수 | 300+ |
| **AI 검색 유입 비중(%)** | 전체 세션 대비 AI 검색 유입 비중 | 10%+ |
| **인용/출처 노출 확인 건수(월)** | AI 검색 결과에서 0to1log가 인용·출처로 노출된 확인 건수 | 20+ |

> **ADR — 왜 GEO를 Phase 3에서 하는가:** GEO는 기존 콘텐츠가 풍부해야 효과가 있다. Phase 1~2에서 콘텐츠와 기본 SEO를 확보한 뒤, Phase 3에서 콘텐츠 구조를 AI 검색에 맞게 고도화한다.

별도의 "바이럴 전략"을 세우기보다, 기존 Phase 3 기능에 바이럴 관점을 녹인다.

## 6. 앱 확장 전략

### 6.1 PWA (Phase 4)

네이티브 앱 전에 PWA(Progressive Web App)를 먼저 배포하여 "앱 경험"을 검증한다.

**PWA 구현 항목:**

| 항목 | 설명 |
|---|---|
| **Web App Manifest** | 앱 이름, 아이콘, 테마 컬러 (Pink!) 설정 |
| **Service Worker** | 오프라인 캐싱 — 저장한 글을 오프라인에서 읽기 |
| **설치 프롬프트** | "홈 화면에 추가" UX |
| **푸시 알림** | Web Push API — 일일 AI 뉴스 알림 (선택적) |

**PWA의 장점:**

- 개발 비용이 거의 제로 — Astro 위에 manifest + service worker만 추가
- 앱 스토어 없이 "앱처럼 설치" 경험 제공
- 네이티브 앱 개발 전 "사람들이 정말 앱을 원하는가"를 데이터로 검증

**측정 지표:**

- PWA 설치율 4%+ (설치 프롬프트 노출 대비)
- 설치자 4주 유지율 25%+
- 푸시 알림 수신 동의율 35%+ / 클릭률 8%+

### 6.2 네이티브 앱 — Expo (Phase 5)

**비전:** AI 뉴스 딜리버리 + 학습 플랫폼

**Phase 5 진입 기준:**

- PWA 설치율 4%+가 4주 연속 유지
- 설치자 4주 유지율 25%+ 달성
- 프리미엄 구독 전환율 2%+ 검증 완료
- 웹 플랫폼이 안정화 상태 (주요 버그 없음, 콘텐츠 파이프라인 안정 운영)

**Expo 앱 핵심 기능:**

| 기능 | 설명 |
|---|---|
| **AI 뉴스 피드** | 웹과 동일한 Daily Dual News + 페르소나 전환 |
| **푸시 알림** | 일일 AI 뉴스 알림, 맞춤 콘텐츠 추천 |
| **오프라인 읽기** | 저장한 글을 오프라인에서 열람 |
| **인앱 결제** | Polar 연동 — 프리미엄 구독 |
| **다크/라이트 모드** | 웹과 동일한 Pink Theme 디자인 시스템 |

**앱 스토어 출시:**

- App Store (iOS) + Google Play (Android)
- Expo EAS Build로 빌드 → 각 스토어 심사 제출

> **ADR — 왜 PWA를 먼저 하는가:** 네이티브 앱은 개발과 유지보수 비용이 크다. PWA로 최소한의 비용으로 "앱 설치 경험"을 먼저 제공하고, 실제 사용 데이터로 네이티브 앱의 필요성을 검증한다. "앱을 만들었는데 아무도 안 쓰는" 상황을 방지하기 위한 전략이다.

## 7. 배제한 항목과 그 이유

명확하게 0to1log에 포함하지 않기로 결정한 항목들:

| 항목 | 배제 이유 |
|---|---|
| **별도 AI API 프로덕트** | 0to1log는 "AI 뉴스 콘텐츠 플랫폼"이지 "사용자가 AI API를 호출하는 SaaS"가 아니다. 기술 스택, 비용 구조, 타겟이 완전히 다르므로 프로젝트 정체성을 흐린다. 하고 싶으면 별도 프로젝트로 분리. |
| **React + Vite 프론트엔드** | Astro + Tailwind로 프론트엔드 스택이 확정됨. 여기에 React + Vite를 섞으면 아키텍처가 꼬인다. 별도 프로젝트에서 사용할 스택. |
| **Phase 4 이전 유료 광고** | 수익 전 광고비 지출은 Solo 프로젝트에서 리스크가 큼. 오가닉 트래픽 확보가 우선. |
