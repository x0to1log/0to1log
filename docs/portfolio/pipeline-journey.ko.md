## 1. 프로젝트 개요

0to1log는 AI/IT 분야의 최신 소식을 자동으로 수집·분류·요약하여 매일 발행하는 뉴스 큐레이션 플랫폼이다. 뉴스에서 등장하는 AI 용어를 자동 추출하여 용어집(Glossary)으로 축적하고, 독자의 수준에 맞춰 Expert/Learner 두 가지 페르소나로 콘텐츠를 제공한다.

### 실제 결과물

실제 다이제스트는 [0to1log.vercel.app](https://0to1log.vercel.app)에서 확인할 수 있다.

### 왜 이 프로젝트를 만들었나

AI 뉴스는 매일 쏟아지지만, 한국어로 된 양질의 기술 브리프는 드물다. 뉴스 사이트들은 보도자료를 그대로 옮기거나, 기술적 맥락 없이 제목만 나열한다. "리서치 엔지니어가 출근길에 읽을 만한 AI 브리프"와 "AI를 처음 접하는 사람도 이해할 수 있는 해설" — 이 두 가지를 하나의 플랫폼에서 자동으로 제공하고 싶었다.

### v1: 근본 원인 발견 (3/10–14, 5일)

첫 5일간 발행 가능한 결과물은 나오지 않았다 — 하지만 시스템을 직접 만들고 테스트하지 않았으면 보이지 않았을 세 가지 아키텍처 결함을 발견했다. 이 결함 각각이 v2의 설계 요구사항이 되었다.

초기 전략은 단순했다: 뉴스 1건을 선택해서 영어로 심층 분석을 쓰고, 한국어로 번역하고, 3개 페르소나(전문가/중급/초급)로 변환한다.

5일 동안 일어난 일:

**1일차–2일차:** 기본 파이프라인 구현. LLM이 5,000자 이상의 글을 안정적으로 생성하지 못하는 문제 발견. 재시도 로직을 추가했다.

**3일차:** EN→KO 번역 시 길이가 50–70%로 줄어드는 문제. 번역 프롬프트에 "원문과 동일한 길이로" 지시를 추가했지만 효과 없음. 품질 기준을 5,000자에서 3,500자로 낮췄다.

**4일차:** 간헐적 JSON 파싱 실패. artifact/resume 시스템을 구현해서 중간 실패 시 이어서 생성할 수 있게 했다. pipeline.py가 979줄에서 1,346줄로 불어났다. 방어 코드만 400줄 이상.

**5일차:** 품질 기준을 2,500자까지 낮췄다. 원래 목표의 50%. "동작은 하지만 품질이 낮은" 상태. 여기서 멈추고 전체를 삭제했다.

```
pipeline.py 크기 변화:
1일차  ████████████████████  979줄
3일차  ██████████████████████████  1,200줄
5일차  ██████████████████████████████  1,346줄 (방어 코드 400줄+)
v2     ████████████  1,127줄 (삭제 후 재작성)
```

**근본 원인:** 잘못된 아키텍처 위에 패치를 쌓고 있었다.

| 증상 | 내가 한 것 (패치) | 근본 원인 |
|------|-------------------|----------|
| KO 번역 길이 부족 | 품질 기준 하향 | EN→KO 순차 번역 자체가 문제 |
| LLM 긴 글 생성 실패 | 재시도 로직 | 1회 호출에 전체 생성이 문제 |
| 중간 실패 | artifact/resume | 파이프라인이 너무 연결되어 있음 |

**발견의 비용:** LLM 호출 $15–25(추정), 발행 가능한 결과물 0. 하지만 여기서 발견한 세 가지 근본 원인 — 순차 번역, 단일 호출 생성, hard validation — 이 v2의 정확한 설계 요구사항이 되었다. 이 단계 없이 v2의 "하루 만에 완성"은 불가능했다.

### v2: 근본 원인을 해결하니 코드가 줄었다 (3/15, 1일)

v1의 세 가지 근본 원인을 직접 공격했다.

| v1 근본 원인 | v2 해결 |
|---|---|
| EN 생성 → KO 번역 (길이 손실) | **EN+KO 동시 생성** (번역 단계 제거) |
| 1회 호출로 전체 생성 (길이 불안정) | **팩트 추출 → 페르소나별 분리 생성** |
| hard validation (파이프라인 사망) | **draft 저장** (admin에서 확인) |

핵심 아키텍처 변경:

```
v1: 뉴스 수집 → 1건 선택 → EN 전체 생성 → KO 번역 → 3 페르소나 변환
v2: 뉴스 수집 → 1건 선택 → 팩트 추출 → 페르소나별 EN+KO 동시 생성
```

**팩트 추출 단계**가 핵심이었다. 원본 기사에서 핵심 사실, 수치, 인용을 구조화된 JSON(FactPack)으로 먼저 뽑아낸다. 그 다음 각 페르소나가 이 FactPack을 기반으로 자기 스타일에 맞게 글을 쓴다. LLM이 "이해 + 글쓰기"를 한 번에 하는 대신, "이해"와 "글쓰기"를 분리한 것이다.

**왜 이 방식을 선택했나:** 대안은 EN→KO 번역 프롬프트를 개선하는 것이었다 — 길이 제약 추가, 짧은 출력 시 재시도 등. 하지만 근본 원인은 아키텍처에 있었다: 번역은 본질적으로 내용을 손실시킨다. 같은 FactPack에서 두 언어를 동시에 생성하면 문제 자체가 사라진다. 증상을 패치하는 대신 원인을 제거한 것이다.

**결과:** 코드가 1/3로 줄었고, 하루 만에 동작했다. v1에서 5일간 쌓은 방어 코드가 전부 불필요해졌다.

### v3: 인프라가 있으니 콘텐츠만 바꾸면 됐다 (3/16, 반나절)

v2가 안정적으로 동작하자 콘텐츠 전략의 한계가 보였다: 하루에 뉴스 1건만 다루는 건 가치가 낮다. AI 뉴스는 하루에도 수십 건이 쏟아지는데, 1건만 골라서 심층 분석하면 나머지를 놓친다.

**결정:** 단일 뉴스 심층 분석 → **데일리 다이제스트** (3–5건 큐레이션)

v2가 만들어놓은 인프라 덕분에 바꿀 것이 적었다:
- `rank_candidates()` → `classify_candidates()` (1건 선정 → 카테고리별 분류)
- Research(논문/모델/오픈소스)와 Business(빅테크/산업/신규도구) 2개 카테고리 분리
- 프롬프트 재작성 (단일 뉴스 분석 → 다이제스트 형태)

파이프라인 골격, 로깅, 에러 처리, DB 스키마는 그대로. **좋은 인프라는 제품 변경을 빠르게 만든다.**

### v4: 2 페르소나 전환과 병렬화 (3/17, 반나절)

두 가지 큰 변경:

**1. 3 페르소나 → 2 페르소나 (Expert + Learner)**

3개 페르소나(Expert/Intermediate/Beginner)를 운영해보니, Intermediate와 Expert의 차이가 모호했다. 독자 관점에서도 3단계 중 어디에 자신이 속하는지 판단하기 어렵다. "나는 어느 정도 아는 사람"은 Expert도 Beginner도 아닌 애매한 위치.

Expert(전문가)와 Learner(학습자) — 이 두 축이 더 명확하다. "깊이 있게 읽고 싶은가, 쉽게 이해하고 싶은가."

- LLM 호출: 6회 → 4회 (카테고리 2 × 페르소나 2)
- 비용: -33%
- UX: 선택지가 줄어 직관적

**왜 3개를 유지하지 않았나:** Intermediate 콘텐츠가 Expert와 70% 이상 겹쳤다. 대안은 Intermediate를 더 차별화하는 것이었지만, 프롬프트 유지 비용이 3배가 되는 대신 UX 가치는 미미했다. "깊이 vs 접근성"이라는 두 개의 명확한 축이 세 개의 모호한 단계보다 독자에게 더 나았다.

**2. 파이프라인 병렬화 — 170초 → 90초 (47% 단축)**

모든 LLM 호출이 순차 실행되고 있었다. 분석해보니 세 가지 병렬화가 가능했다:

| 최적화 | 시간 절감 |
|--------|----------|
| Research + Business 다이제스트 동시 생성 | 약 25s |
| 핸드북 Call 2, Call 3 병렬 | 용어당 약 5s |
| 핸드북 용어 2개씩 동시 생성 | 약 30s |
| **합계** | **약 80s (170s → 90s)** |

비용은 동일하다. 같은 양의 토큰을 사용하되, 대기 시간만 줄인 것.

### v5: 소스 다변화와 품질 체계 (3/18–25, 8일)

v4까지의 파이프라인은 **Tavily 뉴스 검색**에만 의존했다. 이것이 치명적인 편향을 만들었다.

**문제 발견 (3/24):** Research 다이제스트에 실제 연구 논문이 하나도 없었다. "Arm AI 칩 발표", "LG Display 저전력 LCD" 같은 산업 뉴스가 research로 분류되어 있었다.

**근본 원인 3가지:**
1. **수집 소스 편향** — Tavily(뉴스 검색)만 사용. 논문/코드/모델을 직접 수집하지 않아 후보 풀 자체에 research 소재 부족
2. **분류 기준 모호** — "papers: breakthrough results" 같은 넓은 정의. 네거티브 가이드 없음
3. **강제 할당량** — "3–5건 선택" 룰이 기준 미달 기사를 억지로 research에 밀어넣음

**해결:**

**A. 4개 소스 병렬 수집**

| 소스 | 대상 | 일일 후보 |
|------|------|----------|
| Tavily | 일반 AI 뉴스 | 약 15건 |
| HuggingFace Daily Papers | 커뮤니티 큐레이션 논문 | 약 10건 |
| arXiv API | 최신 논문 (cs.AI/cs.CL/cs.LG) | 약 10건 |
| GitHub Trending | ML 오픈소스 리포 | 약 10건 |

후보 풀: 30건 → **45건/일**. `asyncio.gather()`로 병렬 수집.

**B. 분류 프롬프트 강화**
- Research 진입 조건: 기술 산출물(모델 가중치/코드/논문) 기반
- Litmus test: "이 기사의 메인 주제가 모델/코드/논문인가?"
- **0–5 룰**: 기준 미달 시 빈 리스트 허용. 억지 채움 금지

**C. 프롬프트 감사 (52개 이슈)**
전체 프롬프트 파일을 감사하여 52개 이슈를 3단계로 분류:
- **P0 (Critical, 6개):** URL 환각 방지, citation 매핑, 팩트 오류 방지
- **P1 (Important, 18개):** 토큰 효율, few-shot 예시, 점수 정의
- **P2 (Nice-to-have, 28개):** 코드 표준, 구조 개선, 네이밍

**D. 모델 전환: gpt-4o → gpt-4.1**
- IFEval (지시 따르기): 87.4% vs 81%
- 비용: input $2.00/M vs $2.50/M (20% 저렴)
- A/B 테스트 결과: 동일 프롬프트에서 gpt-4.1이 더 일관적

**고려한 대안:** gpt-4o(기존, 익숙), Claude 3.5 Sonnet(지시 따르기 강점), gpt-4.1-mini(저렴). gpt-4o와 gpt-4.1을 A/B 테스트한 결과 — 같은 프롬프트, 같은 실패 패턴이지만 gpt-4.1이 IFEval에서 6% 높고 input 비용이 20% 저렴했다. Claude는 SDK 전환 비용과 기존 PydanticAI + OpenAI 통합 때문에 테스트하지 않았다.

**E. 자동 품질 스코어링**
- 4개 기준 × 25점 = 100점 만점
- Section Completeness, Source Citations, Technical Depth, Language Quality
- gpt-4.1-mini로 평가 (비용 약 $0.004/회)

### v6: 스켈레톤 맵 — 프롬프트 구조의 결정적 전환 (3/26, 1일)

v5까지 프롬프트에 13개 작성 규칙을 나열했다. LLM은 이 규칙들을 부분적으로만 따랐다. 특히 KO 콘텐츠는 EN보다 항상 짧고, 섹션이 누락되었다.

**발견:** 모델을 바꿔도(gpt-4o ↔ gpt-4.1) 같은 패턴으로 실패한다. 모델이 아니라 프롬프트가 문제.

**해결: 페르소나별 스켈레톤 (완성된 예시)**

규칙을 나열하는 대신, 원하는 출력물의 뼈대를 직접 보여줬다. 4개 조합별로 별도의 스켈레톤:

```
SKELETON_MAP = {
    ("research", "expert"):  RESEARCH_EXPERT_SKELETON,
    ("research", "learner"): RESEARCH_LEARNER_SKELETON,
    ("business", "expert"):  BUSINESS_EXPERT_SKELETON,
    ("business", "learner"): BUSINESS_LEARNER_SKELETON,
}
```

**결과:**

| 버전 | EN Biz | EN Res | KO Biz | KO Res | 평균 |
|------|--------|--------|--------|--------|------|
| v1 (규칙만) | 50 | 75 | 60 | 40 | **56** |
| v3 (스켈레톤 1개) | 85 | 90 | 65 | 60 | **75** |
| v6 (스켈레톤 4개) | 95 | 93 | 85 | 88 | **90** |

자동 품질 스코어링(gpt-4.1-mini, 단일 run 최고치):
- **Business: 99점** (Expert 100, Learner 98)
- **Research: 95점** (Expert 95, Learner 95)

### v7: 품질 중심 전면 리팩토링과 롤백의 교훈 (3/28–29, 2일)

v6은 자동화된 품질 점수 90/100을 기록했다. 하지만 실제 발행된 뉴스를 **사용자 관점**에서 읽어보니 평균 76/100이었다.

자동 점수가 잡지 못한 5가지 문제:

| 문제 | 자동 점수 | 사용자 경험 |
|------|----------|----------|

### v8: 구조 분리와 DON'T 제거 (3/29–30, 2일)

v7은 평균 85.3점이었지만, Research Expert가 아이템당 1문단만 작성하는 문제가 지속되었다. 심층 분석으로 세 가지 구조적 원인을 찾았다:

**1. 분류와 랭킹이 한 호출에 결합.** 중요도 점수가 피상적이라 5개 아이템이 동일 깊이로 작성됨.

**2. Research Expert Guide에 DON'T가 너무 많음.** 569단어, 9개 부정 지시. LLM이 "짧게 써라"로 과해석. Business Expert Guide는 201단어, DONT 0개로 90점.

**3. 스켈레톤 placeholder가 잘못된 패턴.** 첫 아이템만 완전 작성, 나머지 `[3 paragraphs...]` → LLM이 나머지를 최소한으로 처리.

**해결:**

**A. 분류/랭킹 분리** — 새 `rank_classified()` 함수(gpt-4.1-mini)가 `[LEAD]`/`[SUPPORTING]` 태그 부여. 비용: run당 $0.00014.

**왜 수정이 아니라 제거인가:** 대안은 9개 부정 지시를 긍정문으로 바꾸는 것이었다. 하지만 Business Expert Guide가 이미 증명하고 있었다 — 더 적은 단어, DONT 0개로 더 높은 점수. 아이템당 1문단이 3문단으로 늘어났다.

**B. Research Expert Guide 리팩토링:** 569 → 151단어, DONT 9 → 0개.

**C. 스켈레톤 2번째 아이템 완전 작성.** **D. Exa 독립 수집기 승격 (5 → 6소스).** **E. Community Pulse 개편 (분위기 요약 + 38개 서브레딧 화이트리스트).**

**v8 품질 결과:** 4개 페르소나 모두 90점 — 모든 조합이 동일한 점수를 받은 첫 번째 버전.

## 3. 핸드북 파이프라인의 진화

핸드북(AI 용어집)은 뉴스 파이프라인과 연동된다. 뉴스에서 등장한 AI 용어를 자동 추출하여, 기초(중학생도 이해 가능)와 심화(시니어 엔지니어 레퍼런스) 두 수준의 해설을 생성한다.

### 초기 → 4-Call 분리 (3/15–17)

**문제:** 하나의 LLM 호출로 16개 필드(KO/EN × basic/advanced × 4섹션)를 한 번에 생성하면, 후반부 필드가 누락되거나 얕아진다. LLM의 토큰 한계.

**해결:** 4단계로 분리:

```
Call 1: 메타 + Basic KO
Call 2: Basic EN        ──┐ 병렬
Call 3: Advanced KO     ──┘
Call 4: Advanced EN
```

Call 2와 Call 3는 독립적이므로 병렬 실행. 용어당 약 5초 절감.

### 10개 용어 유형 분류 (3/18)

**문제:** 모든 용어에 동일한 프롬프트를 사용하니, "BERT"와 "Kubernetes"와 "Funding Round"의 심화 콘텐츠가 비슷한 구조와 깊이를 가졌다. 알고리즘은 수식이 필요하고, 인프라 도구는 아키텍처 다이어그램이 필요하고, 비즈니스 용어는 사례 분석이 필요하다.

**해결:** gpt-4.1-mini로 용어를 10개 유형 중 하나로 분류하고, 유형별 전용 심화 프롬프트를 사용:

| 유형 | 예시 | 심화의 핵심 |
|------|------|-----------|
| 알고리즘/모델 | BERT, Transformer | 수식 유도, 복잡도 분석, 코드 |
| 인프라/도구 | Docker, CUDA | 아키텍처, 설정, 트러블슈팅 |
| 비즈니스/산업 | Funding Round | 시장 데이터, 의사결정 프레임워크 |
| 개념/이론 | Overfitting | 수학적 직관, trade-off |
| 제품/브랜드 | GPT-4, Claude | 경쟁 비교, API 스펙, 벤치마크 |
| 메트릭/지표 | F1, BLEU | 공식 유도, 선택 기준, 오용 사례 |
| 기법/방법론 | Prompt Engineering | 변형 비교, 실패 패턴 |
| 데이터 구조/포맷 | Parquet, ONNX | 내부 구조, 벤치마크 |
| 프로토콜/표준 | OAuth 2.0, gRPC | 핸드셰이크 흐름, 보안 모델 |
| 아키텍처 패턴 | Microservices, RAG | 트레이드오프, 마이그레이션 전략 |

분류 비용: 약 $0.001/용어 (gpt-4.1-mini 1회 호출).

### Tavily 통합 + Self-Critique (3/18)

**문제:** LLM은 knowledge cutoff 이후의 정보를 모른다. "2026년 3월에 발표된 새 모델"에 대한 용어 설명에 최신 정보가 빠진다.

**해결:**
1. **Tavily 검색**: 용어 생성 전 웹 검색으로 최신 정보 5건 수집
2. **Reference Materials로 주입**: 4개 Call 모두에 Tavily 결과를 컨텍스트로 제공
3. **Self-Critique**: 생성 후 gpt-4.1-mini가 "시니어 엔지니어 관점에서 부족한 점" 평가. 점수 75 미만이면 재생성
4. **Quality Scoring**: 4개 기준(깊이/정확성/고유성/완성도) × 25점

### 신뢰도 기반 라우팅 (3/23–25)

뉴스에서 자동 추출된 용어 중 일부는 기술 용어가 아닌 일반 명사였다. "-powered", "-driven" 같은 접미사를 가진 마케팅 표현이 섞여 들어왔다.

**이중 방어:**
1. **접미사 패턴 매칭** (1차): 하드코딩된 패턴으로 빠르게 필터. 비용 0
2. **LLM 2차 필터링**: 남은 용어를 gpt-4.1-mini에게 "진짜 기술 용어만 골라줘". 약 $0.01

**신뢰도 라우팅:**
- High confidence → 자동 생성 (4-call + 품질 체크)
- Low confidence → `status: queued` (사람이 리뷰 후 판단)

## 4. 프롬프트 엔지니어링: 56점에서 90점까지

뉴스 프롬프트 파일(`prompts_news_pipeline.py`) 하나에 **50개 커밋**이 쌓였다. 프롬프트 엔지니어링이 이 프로젝트에서 가장 반복적이고, 가장 많이 배운 영역이다.

### 8회 반복의 기록

| 반복 | 점수 | 핵심 변경 | 배운 것 |
|------|------|----------|---------|
| v1 | **56** | 13개 작성 규칙 나열 | 규칙만으로는 LLM이 따르지 않는다 |
| v2 | **48** | gpt-4o 롤백 (A/B 테스트) | 모델을 바꿔도 같은 패턴으로 실패 → 모델이 아니라 프롬프트가 문제 |
| v3 | **75** | Few-shot 스켈레톤 1개 추가 | LLM은 규칙보다 **완성된 예시**를 정확히 따른다 |
| v4 | **84** | KO 스켈레톤 추가 + 구조 동등성 | "EN 길이의 80%"는 잘못된 기준. 한국어는 같은 내용을 더 짧게 표현 |
| v5 | **84** | 구조 동등성 (섹션/아이템/단락 수) | 글자 수 대신 **구조**(섹션 개수, 아이템 개수)로 검증해야 |
| v6 | **90** | 페르소나별 스켈레톤 4개 분리 | 하나의 스켈레톤을 4개 페르소나가 공유하면 스타일이 섞인다 |
| v7 | **85.3** | 사용자 관점 평가 + 롤백 | 자동 점수는 실제 UX 문제를 놓침; 프롬프트 변경 누적 = 회귀 |
| v8 | **90.0** | DON'T 제거 + 스켈레톤-규칙 일관성 | "하지 마라"는 LLM을 과교정시킴; 스켈레톤이 규칙보다 강하다 |

### 핵심 교훈

**1. "모델이 아니라 프롬프트가 문제"**

v2에서 gpt-4o ↔ gpt-4.1 A/B 테스트를 했다. 두 모델 모두 동일한 패턴으로 실패: 섹션 헤더 무시, citation 누락, KO 콘텐츠 축약. 모델을 바꾸기 전에 프롬프트 구조를 바꿔야 한다.

**2. Few-shot 스켈레톤 > 규칙 나열**

13개 규칙(v1, 56점) vs. 같은 규칙 + 스켈레톤 1개(v3, 75점). LLM은 "이렇게 해라"보다 "이것처럼 해라"를 더 잘 이해한다.

**3. 페르소나별 스켈레톤이 필수**

v3–v5에서 Business Expert 스켈레톤 1개를 4개 조합이 공유했다. Research Learner가 Business Expert처럼 작성되었다 — 비유 대신 전문 용어, 쉬운 설명 대신 밀도 높은 분석. 스켈레톤을 분리하자 Research Learner가 "한 글자씩 순서대로 읽는 대신 페이지 전체를 한 번에 처리한다 — 이를 병렬 diffusion 디코딩이라고 한다" 같은 접근성 높은 표현을 사용하기 시작했다.

**4. KO 품질은 구조 동등성으로 검증**

"EN 길이의 80% 이상"이라는 기준은 한국어에 맞지 않는다. 한국어는 같은 내용을 영어보다 짧게 표현하기 때문이다. "같은 수의 ## 섹션, ### 아이템, 단락"으로 변경하자 KO 커버리지가 EN과 동일해졌다.

**5. Sandwich 패턴: 프롬프트 끝에 체크리스트**

프롬프트 최하단에 FINAL CHECKLIST(8개 검증 항목)를 배치했다. LLM이 출력을 마무리하기 직전에 읽게 되어, citation 형식, 섹션 존재 여부, KO=EN 동등성 등의 준수율이 높아졌다.

**6. 코드 후처리 > 프롬프트 지시**

핸드북 용어 링크를 프롬프트에서 LLM에게 요청하면 정확도가 약 70%. 코드로 후처리하면 100%. "LLM에게 시킬 수 있는 것"과 "코드로 해야 하는 것"의 경계를 아는 것이 중요하다.

## 5. 핵심 기술적 의사결정

### 3-Tier 모델 구조

| 티어 | 모델 | 용도 | 가격 (input/output per 1M) |
|------|------|------|---------------------------|
| Main | gpt-4.1 | 다이제스트 생성, 핸드북 콘텐츠 | $2.00 / $8.00 |
| Light | gpt-4.1-mini | 분류, SEO, 리뷰, 품질 평가 | $0.40 / $1.60 |
| Reasoning | o4-mini | 뉴스 랭킹, 팩트체크 | $1.10 / $4.40 |

gpt-4.1을 선택한 이유: IFEval(지시 따르기) 87.4% — gpt-4o보다 6%p 높고, input 비용 20% 저렴.

### Draft-First 원칙

> "콘텐츠 품질로 파이프라인을 멈추지 않는다. 인프라 에러만 재시도."

품질이 기준 미달이면 draft로 저장하고 admin이 확인한다. 파이프라인 자체는 절대 멈추지 않는다. 이 원칙이 v1에서 v2로의 가장 큰 아키텍처 변화였다.

### 0-5 룰

Research 카테고리에 적합한 뉴스가 없으면 **빈 리스트를 허용**한다. "3–5건 선택"이라는 강제 할당량이 품질을 오히려 떨어뜨렸다 — 기준 미달 기사를 억지로 채워넣게 되기 때문이다.

### 비용 최적화 전략

비용을 줄이기 위해 **하지 않기로 한 것**들도 있다:

| 고려한 것 | 결정 | 이유 |
|-----------|------|------|
| 분류에 gpt-4.1-mini 사용 | 미채택 | 약 $0.03/일 절감이지만 분류 품질 리스크 |
| 품질 체크 제거 | 미채택 | 약 $0.004/일 절감이지만 자동 발행의 선행 조건 |

## 6. 정량적 결과

## 7. 핵심 교훈

### 품질 기준이 내려가면 아키텍처가 잘못된 것이다

v1에서 품질 기준을 5,000자 → 3,500자 → 2,500자로 낮추고 있었다. 이 시점에서 멈추고 재설계해야 했다. v2에서 아키텍처를 바꾸자 v1의 400줄 방어 코드가 전부 불필요해졌다.

### "하지 마라"를 빼면 LLM이 더 잘한다

Research Expert Guide의 DON'T 9개를 전부 삭제했더니 아이템당 1문단 → 3문단. LLM은 부정 지시를 확대 해석한다. Business Expert Guide가 201단어, DONT 0개로 90점을 내고 있었다 — 같은 패턴을 적용한 것.

### 스켈레톤이 규칙을 이긴다

"최소 3문단"을 6곳에 적어도, 스켈레톤에 `[2-3 paragraphs]`가 있으면 LLM이 2문단을 선택한다. 규칙, 스켈레톤, 품질 체크 3곳이 일관되어야 의도대로 동작한다.

### 프롬프트 변경은 한 번에 하나씩

v7에서 3가지 변경을 한 커밋에 넣었더니 점수가 86.5 → 66.5로 폭락. 어떤 변경이 원인인지 분리 불가. "롤백 후 선별 재적용"이 "패치 위에 패치"보다 안전하다.

### LLM의 한계를 인정하고 코드로 보완

핸드북 용어 링크는 프롬프트로 시키면 정확도 약 70%, 코드 후처리는 100%. 태그 정규화, 환각 메타데이터 제거도 코드에서 처리. "LLM이 잘하는 것"(글쓰기, 분류)과 "코드가 잘하는 것"(정확한 매칭, 형식 강제)을 분리하는 감각이 생겼다.

## 8. 현재 아키텍처

### 뉴스 파이프라인 흐름

```
6개 소스 병렬 수집 (Tavily + HuggingFace + arXiv + GitHub + Google RSS + Exa)
    | 50-60건/일
    v
URL 중복 제거 + 발행 이력 제외 (3일) + Google URL 변환 + 필러 필터
    v
분류 (gpt-4.1-mini) --> Research 0-5 / Business 0-5
    v
커뮤니티 수집 (HN Algolia + Reddit JSON) -- 분류된 전체 아이템
    38개 서브레딧 화이트리스트, 분위기 요약 방식
    v
랭킹 (gpt-4.1-mini) -- 카테고리별, engagement 데이터 포함
    --> [LEAD] / [SUPPORTING] 태그 부여
    v
+-- Research 다이제스트 -------+   +-- Business 다이제스트 -------+
|  Expert EN+KO (gpt-4.1)     |   |  Expert EN+KO (gpt-4.1)     |  <-- 병렬
|  Learner EN+KO (gpt-4.1)    |   |  Learner EN+KO (gpt-4.1)    |
+------------------------------+   +------------------------------+
    | [LEAD] 3-4문단, [SUPPORTING] 3+문단
    | max_tokens 32K, Community Pulse 데이터 있으면 필수
    v
후처리: bold fix + [LEAD]/[SUPPORTING] 태그 제거
    v
품질 스코어링 (o4-mini, 4개 프롬프트: R/B x Expert/Learner)
    v
draft 저장 --> admin 확인 --> 발행
    v
핸드북 용어 자동 추출 (조건부)
```

### 핸드북 용어 생성 흐름

```
용어 입력
    |
    +-- Tavily 검색 (5건)        --+  병렬
    +-- gpt-4.1-mini 유형 분류   --+
                |
                v
    유형별 전용 심화 프롬프트 선택
                |
    +-------------------------------+
    | Call 1: 메타 + Basic KO       |
    | Call 2+3: Basic EN // Adv KO  |  <-- 병렬
    | Call 4: Advanced EN           |
    +-------------------------------+

    score < 75 --> 재생성
                |
    Quality Check (4기준 x 25점)
    score < 60 --> warning
                |
                v
    저장 (draft / queued)
```
