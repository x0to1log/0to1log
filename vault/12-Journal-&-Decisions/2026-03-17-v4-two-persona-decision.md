# 결정: 3 페르소나 → 2 페르소나 전환 (v4)

> 날짜: 2026-03-17
> 맥락: v3 다이제스트 페르소나 차별화 실패 분석 후 전환 결정

---

## 전환점

v3에서 3 페르소나(Expert/Learner/Beginner)를 운영하면서 퀄리티를 분석했을 때, **Learner와 Beginner 출력이 사실상 동일**하다는 것을 발견.

### 시도한 것
1. **톤 기반 차별화** (v3 초기) → 실패. "전문적으로 써라" vs "쉽게 써라" — LLM이 비슷하게 출력.
2. **"읽은 후 행동" 축 차별화** (v3 개선) → Expert는 성공. Learner/Beginner는 여전히 실패.
   - Expert: "전략적 판단" 섹션 포함, 조건부 추천, 경쟁 역학 분석 → **명확히 다른 글**
   - Learner: "우리 팀에 미치는 영향" + "지금 할 수 있는 것"
   - Beginner: "나에게 왜 중요한가" + "더 알아보기" → **실제 출력이 Learner와 거의 동일**
3. **DB 확인**: `content_learner = content_beginner`는 `false` (텍스트는 다름) — 하지만 **내용과 구조가 거의 동일**. 섹션 헤더마저 Learner 것을 사용.

### 왜 실패하는가

LLM 입장에서 Learner("업무에 적용")와 Beginner("이해하고 학습")의 차이가 너무 미묘:
- 같은 뉴스 입력
- 같은 카테고리 구조 (Big Tech / Industry / New Tools)
- 둘 다 "비전문가에게 설명하는" 관점
- LLM은 자연스럽게 중간 수준으로 수렴

반면 Expert는 완전히 다른 관점(의사결정/전략)이라 차별화가 작동.

## 결정

**2 페르소나: Expert (현직자) + Learner (학습자)**

| 페르소나 | 구 v3 | 신 v4 |
|---------|-------|-------|
| **Expert (현직자)** | 전략 분석, 의사결정 | 유지 — 잘 작동함 |
| **Learner (학습자)** | 실무 적용 | Learner + Beginner 통합: 이해 + 적용 + 학습 |
| **Beginner (입문자)** | 쉬운 설명, 핸드북 링크 | **삭제** — Learner에 흡수 |

### 왜 Expert + Learner인가 (Expert + Beginner가 아니라)
- "학습자"가 "입문자"보다 넓은 대상을 포괄
- 학습자에 핸드북 링크 + 비유 + 실무 적용을 모두 포함 가능
- "현직자 / 학습자" 라벨이 직관적

## 교훈

1. **LLM 차별화는 "관점"이 달라야 작동한다.** 톤이나 난이도 차이로는 부족. Expert(의사결정)처럼 글의 목적 자체가 달라야 LLM이 다른 글을 쓴다.
2. **3개보다 2개가 나을 때가 있다.** 차별화 안 되는 3개는 독자에게 혼란. 명확한 2개가 더 가치 있다.
3. **프롬프트 한계를 인정해야 한다.** 아무리 정교한 프롬프트도 LLM의 "자연스러운 수렴"을 이길 수 없을 때가 있다. 그때는 구조를 바꿔야 한다.
4. **데이터로 판단한다.** "이론적으로 차별화 가능"이 아니라 "실제 출력이 다른가"로 판단.

## 비용 영향

### 계산 근거
- 모델: gpt-4o (input $2.50/1M tokens, output $10.00/1M tokens)
- 페르소나당 평균 사용량: input ~4000 tokens, output ~7000 tokens (pipeline_logs 실측 기반)
- 페르소나당 비용: (4000 × $2.50 + 7000 × $10.00) / 1M = $0.01 + $0.07 ≈ **$0.08/호출**

### 비용 비교

| | v3 (3 페르소나) | v4 (2 페르소나) | 절감 |
|---|---|---|---|
| 다이제스트 LLM 호출 | 3 × 2타입 = 6회 | 2 × 2타입 = 4회 | -2회 |
| 다이제스트 비용 | 6 × $0.08 = $0.48/일 | 4 × $0.08 = $0.32/일 | -$0.16/일 |
| 퀄리티 체크 | 2회 × ~$0.002 | 동일 | 무시 가능 |
| **일일** | **~$0.48** | **~$0.32** | **-33%** |
| **월간** | **~$14.40** | **~$9.60** | **-$4.80** |
| **연간** | **~$172.80** | **~$115.20** | **-$57.60** |

### 비용 외 효과
- 파이프라인 실행 시간 ~30% 단축 (LLM 호출 2회 감소)
- 코드 단순화 (beginner 관련 코드 전부 삭제)
- 페르소나 차별화 명확 (2개가 3개보다 나음)
- **비용 줄이면서 퀄리티 올린 케이스**

## EN/KO 분리 호출 — 검토 후 미채택

### 배경
v3에서 KO 누락 문제 발생 (beginner 33% 실패율). EN/KO를 분리 호출하면 누락 0%이지만:

### 미채택 이유
1. **내용 불일치**: EN과 KO가 서로의 출력을 모르고 독립 생성 → 강조점, 분석 각도가 달라짐
2. **번역투 위험**: EN을 context로 넘기면 KO가 번역체가 됨 → 퀄리티 저하
3. **문제 해소**: beginner 삭제 후 KO 누락이 주로 발생하던 페르소나가 사라짐. expert/learner에서는 KO 누락 데이터 0%.
4. **안전망 존재**: 혹시 누락 시 KO recovery 호출이 자동으로 재생성

### 결론
현재 구조(EN+KO 동시 생성 + recovery 안전망) 유지. 추가 비용 없음.
