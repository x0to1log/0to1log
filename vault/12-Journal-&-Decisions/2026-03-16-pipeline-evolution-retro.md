# 회고: News Pipeline v1 → v2 → v3 진화

> 같은 문제를 세 번 풀었다. 매번 더 빠르고, 더 적은 코드로, 더 나은 결과를 냈다.

---

## 타임라인 요약

| | v1 | v2 | v3 |
|---|---|---|---|
| **기간** | 5일 (3/10~14) | 1일 (3/15) | 반나절 (3/16) |
| **결과** | 실패 → 전체 삭제 | 동작 → 안정 운영 | 설계 + 구현 완료 |
| **코드** | ~3,444줄 (백엔드) | ~1,127줄 | ~1,127줄 (v2 리팩터) |
| **테스트** | ~3,094줄 (삭제) | 안정 | 유지 |
| **비용/일** | $0.42~1.26 (재시도 포함) | ~$0.43 | ~$0.59 |
| **LLM 호출/실행** | 6회 | 4회 | 6회 |
| **콘텐츠** | 단일 뉴스 심층 분석 (실패) | 단일 뉴스 3페르소나 | 3~5건 데일리 다이제스트 |

---

## 왜 점점 빨라졌나

### v1: 5일 — 잘못된 방향으로 열심히 달림

문제를 코드로 해결하려 했다. LLM이 긴 글을 못 쓰면 재시도 로직을 추가하고, 번역 길이가 부족하면 임계값을 낮추고, 중간에 실패하면 artifact/resume 시스템을 만들었다.

5일간 일어난 일:
- pipeline.py: 979줄 → 1,346줄 (방어 코드만 400줄+)
- 품질 기준: 5,000자 → 2,500자 (50%로 하락)
- 개발 비용: 추정 $15~25 (수십 회 디버깅 실행)

**교훈:** 패치를 쌓는 속도가 아니라, 멈추고 재설계하는 판단이 속도다.

### v2: 1일 — 근본 원인을 해결하니 코드가 줄었다

v1의 세 가지 근본 원인을 직접 공격:

| v1 근본 원인 | v2 해결 |
|---|---|
| EN 생성 → KO 번역 (길이 손실) | EN+KO 동시 생성 (번역 단계 제거) |
| 1회 호출로 전체 생성 (길이 불안정) | 팩트 추출 → 페르소나별 분리 생성 |
| hard validation (파이프라인 사망) | draft 저장 (admin에서 확인) |

코드가 1/3로 줄었고, 하루 만에 동작했다. v1에서 5일간 쌓은 방어 코드가 전부 불필요해졌다.

**교훈:** 좋은 아키텍처는 방어 코드를 제거한다.

### v3: 반나절 — 인프라가 있으니 콘텐츠만 바꾸면 됐다

v2가 만들어놓은 것들:
- 수집 → 분류 → 생성 → 저장 파이프라인 인프라
- 스테이지별 로깅 + 비용 추적
- 에러 처리 + retry + draft 저장
- 프론트엔드 페르소나 탭 시스템

v3에서 바꾼 것은 "콘텐츠 전략"뿐이었다:
- `rank_candidates()` → `classify_candidates()` (1건 선정 → 카테고리별 3~5건 분류)
- 프롬프트 재작성 (단일 뉴스 분석 → 다이제스트 형태)
- `_generate_post()` → `_generate_digest()` (오케스트레이터 전환)

파이프라인 골격, 로깅, 에러 처리, DB 스키마는 그대로. **인프라가 안정적이면, 제품 변경이 빠르다.**

---

## 개발 속도 그래프

```
v1 ████████████████████████████████████████ 5일 (실패)
v2 ████████         1일 (성공)
v3 ████             반나절 (성공)
```

5일 → 1일 → 반나절. 매 이터레이션마다 기반이 쌓이면서 다음 버전이 기하급수적으로 빨라졌다.

---

## 비용 효율

| | 개발 시간 | 개발 비용 (LLM) | 일일 운영 비용 | 콘텐츠 가치 |
|---|---|---|---|---|
| v1 | 5일 | $15~25 | 동작 안 함 | 없음 |
| v2 | 1일 | ~$2 | $0.43 | 뉴스 2건/일 |
| v3 | 반나절 | ~$1 | $0.59 | 뉴스 6~10건/일 |

v1에서 $15~25를 태우고 결과물이 0이었던 것 vs. v3에서 $1 미만으로 6~10건 뉴스 다이제스트를 만들게 된 것. **10배 이상의 비용 효율 개선.**

---

## 핵심 교훈 3가지

1. **"더 열심히"가 아니라 "다르게"** — v1은 5일간 열심히 패치했지만 실패. v2는 아키텍처를 바꾸니 하루 만에 성공.

2. **좋은 인프라는 반복을 가속한다** — v2에서 로깅, 에러 처리, 비용 추적을 제대로 만들어두니, v3는 프롬프트와 오케스트레이터만 바꾸면 됐다.

3. **빠른 포기 = 빠른 진행** — v1을 3일째에 버렸으면 v3까지 이틀 더 빨리 도착했을 것. 임계값을 낮추기 시작한 순간이 포기해야 할 순간이었다.
