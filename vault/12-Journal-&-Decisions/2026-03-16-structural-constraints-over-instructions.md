# 구조적 제약 > 지시문

## 발단

핸드북 용어 생성 프롬프트는 한 번의 LLM 호출로 EN+KO 양쪽 필드를 동시에 채우는 구조였다. 처음에는 당연해 보였다 — 한 번에 다 뽑으면 빠르고 간단하니까.

문제는 금방 나왔다. 영어 섹션 안에 한국어 헤더가 들어오고, 한국어 본문에 영어 용어가 섞였다. 모델이 언어 경계를 지키지 않는 것이다.

## "더 잘 지시하면 되지 않나" — 실패한 접근

처음 든 생각은 지시문을 강화하는 것이었다.

- "NEVER mix languages"
- "Output Korean ONLY in korean fields"
- "English content must contain zero Korean characters"

프롬프트가 길어질수록 모델이 더 잘 따를 것 같았다. 실제로는? 가끔 나아지다가 다시 섞였다. 재현 불가능한 품질. 프롬프트를 아무리 다듬어도 "대체로 지킴" 수준을 벗어나지 못했다.

## 실제 해결책 — 구조로 틀어막기

결국 접근법 자체를 바꿨다.

1. **JSON 키 분리**: `definition_en`, `definition_ko`, `body_en`, `body_ko`처럼 언어별로 키를 명시적으로 나눴다. 모델 입장에서는 각 키에 무엇을 채워야 하는지 구조 자체가 말해준다.

2. **LLM 호출 분리**: 단일 호출을 3개로 쪼갰다 — meta+Basic(EN), meta+Basic(KO), Advanced. 각 호출의 시스템 프롬프트는 단일 언어 컨텍스트만 가진다. 한국어 호출에 영어가 섞일 구조적 여지가 없어진다.

결과는 명확했다. 지시문을 한 줄도 추가하지 않았는데 언어 혼용이 사라졌다. → [[Handbook-Prompt-Redesign]]

## 뉴스 파이프라인에서 같은 교훈

[[AI-News-Pipeline-Design]]에서도 동일한 패턴이 있었다.

- **v1**: "영어로 생성한 뒤 한국어로 번역하라" — 지시 기반. 번역 품질이 들쭉날쭉하고 두 언어가 구조적으로 연결되지 않았다.
- **v2**: 단일 JSON에 `title_en` / `title_ko` / `summary_en` / `summary_ko`를 동시 생성. 번역 단계가 사라지고, 두 언어가 같은 컨텍스트에서 나오니 일관성이 올라갔다.

지시가 아니라 스키마가 동작을 제어한 것이다.

## 핵심 인사이트

> "잘 지시해줘"보다 "잘못된 출력이 구조적으로 불가능하게" 만드는 게 낫다.

LLM은 확률적 시스템이다. 지시문은 확률 분포를 조금 밀 뿐이다. 구조(JSON 스키마, 호출 분리, 컨텍스트 격리)는 잘못된 출력이 나올 공간 자체를 없앤다.

이 차이는 크다. 지시문 기반 제약은 테스트할수록 엣지 케이스가 나온다. 구조적 제약은 엣지 케이스 자체가 성립하지 않는다.

## 앞으로 적용할 기준

- LLM 출력에서 "절대 ~하면 안 됨" 류의 지시가 필요해진다면 → 먼저 구조로 풀 수 있는지 확인한다.
- 단일 호출에 두 가지 언어/포맷/도메인을 섞는 설계는 일단 의심한다.
- JSON 스키마 키 이름 자체가 모델에게 힌트다 — 키를 잘 설계하는 것도 프롬프트 엔지니어링이다.
