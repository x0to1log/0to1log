# 회고: 프롬프트 튜닝은 반복 작업이다

> 날짜: 2026-03-16
> 맥락: v3 다이제스트 + 핸드북 파이프라인 첫 실행 후 퀄리티 문제 발견

---

## 오늘 하루 요약

하루 동안 뉴스 파이프라인 v2→v3 전환, 핸드북 프롬프트 아키텍처 v2, 파이프라인 컨트롤(Cancel, 타임아웃, skip_handbook), 에디터 UX 개선 등 대규모 작업을 수행함.

### 잘 된 것
- **아키텍처**: 파이프라인 분리(News/Handbook), 3회 호출 분리, 스테이지별 로깅 — 구조적으로 견고
- **인프라**: Cancel 버튼, Stuck 타임아웃, Retry Handbook, 비용 추적 — 운영 도구 완비
- **코드 품질**: 57개 테스트 전체 통과, ruff 클린 유지

### 반복 수정이 필요했던 것 (정상)
- **프롬프트 튜닝**: self-check → 3단어 제한 → section-per-key → 3회 호출 분리
  - 이건 "코드를 계속 고치는 것"이 아니라 **프롬프트 엔지니어링의 본질적 반복**
  - 한 번에 완벽한 프롬프트는 없다 → 실행 → 결과 확인 → 조정 사이클
- **버그 수정**: NOT NULL 위반, RLS 정책, run_key UNIQUE 충돌
  - 배포 후에만 발견되는 인프라 이슈 — 로컬 테스트로는 안 잡힘

### 배운 교훈
1. **프롬프트에서 해결 vs 코드에서 해결** — 핸드북 링크를 프롬프트에서 요구하는 것보다 코드 후처리가 100% 확실
2. **구조적 강제 > 지시 강화** — "8개 섹션 다 써줘" 체크리스트보다 JSON 키 분리가 더 효과적
3. **LLM 토큰 한계 인식** — 16개 키를 한 번에 출력하면 후반부 누락 → 호출 분리 필요
4. **배포 환경 차이** — 로컬 테스트 통과 ≠ 프로덕션 동작 (RLS, 마이그레이션, 타이밍)

### 현재 남은 프롬프트 튜닝
- KO 다이제스트 제목이 영어로 나옴 → LLM이 headline_ko 생성하게
- 용어 추출에서 형용사/일반 개념이 여전히 포함됨 → 필터 규칙 더 강화
- Few-shot 예시 부재 → 좋은 출력물 확보 후 추가 예정
